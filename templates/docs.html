Quickstart

  1. Upload your resumeA PDF is parsed into structured sections: contact, summary, experience, education, skills, projects.
  2. Review the structureFix anything the parser got wrong. Content can be polished later.
  3. Paste a JDDrop in a job description, a posting URL, or search by company and role.
  4. GenerateThe pipeline rewrites, verifies ATS fit, and iterates until it scores well and fits on one page.
  5. DownloadExport the PDF. Every save is versioned so you can roll back at any time.

How the pipeline works

Pasting a JD triggers a seven stage pipeline. Each stage is swappable and logs its output so you can see what changed and why.

  1. Pre-screen A local check (no LLM call) that the JD looks like a real job description, flagging obvious mismatches early.
  2. Version match Looks for an existing saved resume version tagged for a similar role, and reuses it if the overlap is strong.
  3. Generator The LLM rewrites your resume YAML targeting the JD, keeping factual bullets but reordering and rephrasing for relevance.
  4. PDF fit Renders the YAML to PDF via WeasyPrint. If it spills past one page, it compresses and re-checks. Everything happens in memory.
  5. ATS verify A separate LLM call scores the rendered resume against the JD on keyword coverage, seniority alignment, and structure.
  6. Improvement loop If the score is under 90, or the PDF is over one page, the loop tightens and re-verifies. It runs up to a few iterations.
  7. Tag and save The final version is saved with tags derived from the JD (company, role, seniority) so the version match stage can find it next time.

JD input formats

  • Text Paste the job description as-is. The pipeline handles long, messy, copy pasted postings.
  • URL Paste a job posting URL. We fetch, strip boilerplate, and extract the JD text server-side.
  • Search Type a company and role (for example, "Stripe senior backend engineer") and we'll find the posting.
Some sites (LinkedIn, Greenhouse behind auth walls) will block the fetch. If a URL fails, paste the JD text instead.

Styling and versioning

  • Style presets. Font family, size, line height, margins, and accent color are all adjustable in Settings. Changes render live in the editor preview.
  • Version history. Every save creates a version. The History panel in the editor lets you view, diff, and restore any prior version.
  • Custom sections. The parser detects non-standard sections (publications, awards, certifications) and preserves them. You can rename or reorder from Settings.

Privacy and data handling

Your resume is personal data. We treat it that way. The goal of this product is to keep your content under your control, and to keep the surface area of where it travels as small as possible.

What we store, and where

Your content lives in Postgres

Resume YAML, cover letter drafts, markdown rule databases, and resume learning logs are stored in a single Postgres database, scoped to your user account. Nothing is written to disk between requests.

PDFs are ephemeral

PDFs are rendered in memory by WeasyPrint at the moment you download or preview, then streamed to your browser. No rendered PDF is ever persisted on the server.

API keys are encrypted at rest

If you bring your own LLM provider key, it is encrypted with a server side Fernet key before being written to Postgres, and decrypted only when a request needs to call the provider.

Auth and session isolation

Every read and write is scoped to your authenticated user id. Your resume data, versions, and settings are never visible to other users or accounts.

In transit

Traffic to the app is served over HTTPS. Session cookies are signed with the server's secret key and are HTTP only, so they can't be read by browser scripts.

No third-party analytics

There are no tracking pixels, ad networks, or behavioral analytics on this site. We don't share your content with any third party outside the LLM calls described below.

What leaves the server

When the pipeline runs, relevant excerpts of your resume and the pasted JD are sent to the LLM provider you selected (for example, Anthropic or OpenAI) so the model can do its work. That call is governed by the provider's data policy, not ours. A few things to know:

  • We only send what the stage needs: the resume YAML, the JD text, and the prompt scaffolding around it. No credentials, no database dumps, and no other users' data.
  • If you bring your own API key, the request is billed to your account with the provider, and the provider treats the data according to their retention and training policies.
  • No background jobs call the LLM on your content. LLM calls happen only when you click a pipeline action (Generate, Analyze, Improve).

What we never do

  • We don't train any model on your resume, your cover letters, or your prompts.
  • We don't sell, rent, or share your data with marketers or data brokers.
  • We don't log full request or response bodies from the LLM in a way that is tied to your account beyond what is needed to show you the last pipeline run.

Deletion and export

Deleting your account (Settings, or the user menu in the top right) permanently removes your resume YAML, cover letter drafts, rule databases, version history, pipeline logs, and encrypted API keys from Postgres. There is no soft delete. Once the row is gone, it is gone. If you just want to start from a blank slate without deleting the account, you can wipe individual documents from Settings.

YAML and markdown sources are the source of truth, so you can always copy them out of the editor to keep a local backup.

FAQ

Can I bring my own API key?

Yes. In Settings, under AI Configuration, pick a provider, paste a key, and choose a model. Keys are encrypted with a Fernet key on the server, and decrypted only when a request calls the provider.

What happens if the parser misreads my resume?

You can edit the structure directly after upload. The structure-first review step is designed for exactly this: fix section boundaries and field mappings before any LLM work runs.

Does the pipeline hallucinate experience?

The generator is prompted to preserve factual content and only rephrase or reorder. If a JD requires a skill you don't have, it won't be fabricated. You'll see lower ATS coverage instead.

Can I export formats other than PDF?

Not yet. The source of truth is YAML and rendering is WeasyPrint only today. DOCX export is a possible future addition.

Where is my data stored physically?

In the Postgres database configured for the deployment. On the default deployment that's a managed Postgres instance. If you self host, your data stays entirely within the database you provision.

What gets deleted when I delete my account?

Everything tied to your user id: resume YAML, cover letters, rule databases, version history, pipeline logs, and encrypted API keys. The delete is immediate and permanent.
